# VSeed

Reference material for `packages/vseed`, VSeed examples, generated VSeed website
pages, option documentation, API documentation, and the source-to-docs generation
workflow.

## Sources of Truth

- Runtime and lowering code: `packages/vseed/src/**`.
- Public DSL types: `packages/vseed/src/types/**`.
- Runnable examples: `packages/vseed/tests/examples/**`.
- Generated example tests: `packages/vseed/tests/examples/**/*.test.ts`.
- Generated website docs: `apps/website/docs/zh-CN/vseed/**`.
- API documentation generator: `packages/vseed/scripts/build-api.mjs`.

Do not start by hand-editing generated VSeed documentation or generated tests.
First modify the source types, examples, or generator, then regenerate.

## Workflow

1. Modify the source file that owns the behavior.
2. When a user-visible DSL changes, add or update examples at the same time.
3. Regenerate:

```bash
pnpm --filter @visactor/vseed g
```

4. Validate:

```bash
pnpm --filter @visactor/vseed run test
pnpm --filter @visactor/vseed run typecheck
pnpm --filter @visactor/vseed run lint
```

When only examples change, run at least the affected generated tests.

## Examples

Example JSON files live in:

- `packages/vseed/tests/examples/chartType/<chartType>/*.json`
- `packages/vseed/tests/examples/features/<feature>/*.json`

Each example must include:

- `name`
- `description`
- `vseed`

`build:examples` generates MDX into
`apps/website/docs/zh-CN/vseed/examples`.

`build:test` regenerates aggregate tests, for example:

- `packages/vseed/tests/examples/chartType/chartType.test.ts`
- `packages/vseed/tests/examples/features/features.test.ts`

If generated tests require manual import edits, fix the generator first.

## Types and Documentation

When editing `packages/vseed/src/types/chartType/**`:

- Keep interface names aligned with chart directories where possible.
- Write useful JSDoc for public DSL properties.
- Include `@description`, `@example`, and necessary domain notes for
  user-visible fields.
- Export referenced property types so the documentation generator can resolve
  them.
- Avoid directly modifying `apps/website/docs/zh-CN/vseed/option/*.md`.

`build:docs` generates option documentation from TypeScript interfaces.

## Generated Diff Checks

After running `pnpm --filter @visactor/vseed g`, check the expected generated
changes:

- `packages/vseed/tests/**/*.test.ts`
- `apps/website/docs/zh-CN/vseed/examples/**`
- `apps/website/docs/zh-CN/vseed/option/**`
- API documentation generated by `build-api.mjs`

If the generator touches unrelated files that were already dirty, identify the
reason and report the scope instead of blindly reverting.
