# 1. 这份指南解决什么问题

你要做的是一个给 agent 学习“绘制图表构建页面”的 UI 指南。这个页面不是普通后台管理页，也不是单纯展示一个图表，而是一个 **Chart Builder / BI Editor / 数据可视化工作台**。

它的核心任务是：

- 让用户加载数据；
- 看到 schema 字段；
- 选择 dimension / measure；
- 配置图表类型；
- 配置 encoding slot；
- 添加 WHERE / HAVING filter；
- 修改 limit、theme、locale；
- 渲染图表；
- 在 loading、empty、error、rendered 状态之间稳定切换。

因此，agent 不应该把页面画成：

```text
一堆表单 + 一个图表容器 + 几个按钮
```

而应该把它画成：

```text
数据入口 → 字段选择 → 图表配置 → 筛选条件 → 图表预览 → 结果反馈
```

这是一条完整的用户操作路径。

# 2. 先建立正确的 UI 思维

## 2.1 不要从组件开始，要从用户问题开始

用户打开这个页面时，心里想的不是：

```text
我要操作 VBIChartBuilder。
```

而是：

```text
我有什么数据？
我能用哪些字段？
我想画什么图？
我把哪个字段放到 X 轴？
我把哪个指标放到 Y 轴？
我能不能筛选一部分数据？
为什么现在没有图？
为什么图表报错？
```

所以页面必须回答这些问题：

| 用户问题                    | 页面对应区域                         |
| --------------------------- | ------------------------------------ |
| 我现在连了什么数据？        | Data Source / Toolbar                |
| 我有哪些字段？              | Field Panel                          |
| 哪些是维度，哪些是指标？    | Field Panel 分组                     |
| 当前图表用了哪些字段？      | Shelves / Slots                      |
| WHERE / HAVING 条件是什么？ | Filter UI                            |
| 图表现在是什么状态？        | Chart Workspace                      |
| 出错了怎么办？              | Error fallback / Retry / View detail |
| 我怎么切换展示模式？        | View / Edit mode                     |

## 2.2 图表构建页面的本质是“映射”

图表不是直接从数据变出来的，而是通过映射产生：

```text
字段 Field
  ↓
角色 Role：dimension / measure
  ↓
位置 Slot：X / Y / Color / Size / Label / Tooltip
  ↓
Query / Aggregation / Filter
  ↓
Visual Encoding
  ↓
Chart Result
```

所以页面必须让用户看见这条链路。

好的页面会让用户一眼看懂：

```text
Category 被放到了 X 轴
SUM(Sales) 被放到了 Y 轴
Region 被放到了 Color
WHERE 过滤了 Region = East / West
HAVING 过滤了 SUM(Sales) > 1000
```

差的页面只会显示：

```text
一些 dropdown
一些 input
一个图表
```

用户完全不知道图为什么这样生成。

# 3. 四个参考产品分别学什么

| 产品     | 最核心的 UI 思想                    | VBI 应该学习的部分                                         |
| -------- | ----------------------------------- | ---------------------------------------------------------- |
| Tableau  | 字段拖拽到 shelves/cards 后形成视图 | Field Panel、Shelves、Marks/Encoding、Status Bar           |
| DataWind | 现代 BI 图表表达、主题、交互、叙事  | Chart Gallery、Theme Studio、Interaction Layer、Annotation |
| Superset | Query 配置和 Customize 配置分离     | Query Panel、Customize Panel、Run Query 模式               |
| Hex      | 分析过程复杂，但发布成干净 Data App | View/Edit Mode、Control Strip、Insight Cards、App Layout   |

一个优秀的 VBI 页面不应该照抄某一个产品，而应该组合它们的优点：

```text
Tableau：教会 agent 怎么组织字段和映射
Superset：教会 agent 怎么组织配置面板
DataWind：教会 agent 怎么做现代 BI 视觉与交互
Hex：教会 agent 怎么把复杂编辑器收敛成干净应用
```

# 4. Tableau 风格模块：专业分析工作台

## 4.1 Tableau 的核心气质

Tableau 的页面像一个专业分析工作台。它不是先让用户填表，而是让用户从左边 Data Pane 拿字段，然后放到 Rows、Columns、Marks、Filters 等 shelves/cards 上，图表画布随之变化。

这种设计的核心是：

```text
字段在哪里，图表结构就在哪里产生。
```

这对 VBI 非常重要。因为你的页面也有：

- dimension；
- measure；
- X / Y / Color / Size；
- WHERE；
- HAVING；
- chart workspace。

## 4.2 Tableau 风格适合 VBI 的模块

### 模块 A：Data Pane / 字段浏览器

字段面板不要做成普通列表。它应该像“数据词典 + 可拖拽字段仓库”。

```text
Field Panel
├── Search fields...
├── Dimensions
│   ├── Aa  Category        string
│   ├── Aa  Region          string
│   └── 📅  Order Date      date
├── Measures
│   ├── #   Sales           number
│   ├── #   Profit          number
│   └── #   Quantity        number
└── Derived Fields / Calculated Fields
```

设计细节：

- 维度字段用偏冷色，例如蓝色、紫色；
- 指标字段用偏暖色，例如绿色、橙色；
- 字段名长时必须省略；
- hover 时显示 tooltip：字段类型、示例值、是否可聚合；
- 字段过多时使用分组、搜索、折叠；
- professional 版本支持拖拽；
- minimalist 版本支持点击添加即可。

### 模块 B：Shelves / 字段放置区

Tableau 的 rows/columns 思路可以迁移成 VBI 的 shelves：

```text
Shelves
├── Dimensions  [Category ×] [Region ×]
├── Measures    [SUM(Sales) ▼ ×]
├── WHERE       [Region = East/West ×]
└── HAVING      [SUM(Sales) > 1000 ×]
```

这个区域的作用不是“好看”，而是让用户知道：

```text
当前图表到底由哪些字段和条件生成。
```

视觉建议：

- shelf 使用浅色卡片；
- 每个 token 是圆角 pill；
- dimension token 和 measure token 颜色不同；
- token 内部显示聚合方式，例如 `SUM(Sales)`；
- token 支持删除、排序、打开菜单；
- 空 shelf 用虚线框和提示文字。

### 模块 C：Marks / Encoding Card

Tableau 的 Marks Card 可以迁移成 VBI 的 Encoding Slots：

```text
Visual Mapping / Encoding
├── X Axis       Category
├── Y Axis       SUM(Sales)
├── Color        Region
├── Size         Profit
├── Label        Sales
├── Tooltip      Category, Sales, Profit
└── Detail       Customer Segment
```

这个模块适合 professional 页面。它比普通表单更直观，因为它告诉用户：

```text
字段不是随便选的，而是被映射到视觉通道上。
```

## 4.3 Tableau 风格页面结构示意

```text
┌──────────────────────────────────────────────────────────────┐
│ Toolbar: Data Source | Undo | Redo | Save | Refresh | View   │
├───────────────┬──────────────────────────────────────────────┤
│ Field Panel   │ Shelves                                      │
│               │ ┌──────────────────────────────────────────┐ │
│ Search        │ │ X / Columns: Category                    │ │
│ Dimensions    │ │ Y / Rows: SUM(Sales)                     │ │
│ Measures      │ │ Color: Region                            │ │
│               │ │ Filters: Region = East                   │ │
│               │ └──────────────────────────────────────────┘ │
│               ├──────────────────────────────────────────────┤
│               │ Chart Canvas                                 │
│               │                                              │
│               │                 Bar Chart                    │
│               │                                              │
│               ├──────────────────────────────────────────────┤
│               │ Status: 500 rows · 12 marks · updated now    │
└───────────────┴──────────────────────────────────────────────┘
```

## 4.4 什么时候采用 Tableau 风格

适合：

- 字段选择和拖拽是核心操作；
- 用户需要频繁试不同维度和指标；
- 用户希望像专业 BI 工具一样探索数据；
- 页面有足够空间显示字段和 shelves。

不适合：

- 极简 demo；
- 手机端；
- 只需要固定图表展示的应用页面。

# 5. DataWind 风格模块：现代 BI 视觉与图表叙事

## 5.1 DataWind 的核心气质

DataWind 更像现代 BI 产品和大屏/仪表盘工具的结合。它重视：

- 丰富图表类型；
- 主题和配色；
- tooltip、drill、annotation 等交互；
- 数据门户、驾驶舱、报表、大屏等视觉表达。

它给 VBI 的启发是：

```text
图表构建页面不能只让图“出现”，还要让图“可读、可探索、可讲故事”。
```

## 5.2 DataWind 风格适合 VBI 的模块

### 模块 A：Chart Gallery / 图表类型库

不要只用一个普通下拉框选择图表类型。更好的方式是图表类型库：

```text
Chart Gallery
├── Recommended
│   ├── Bar
│   ├── Line
│   └── Table
├── Comparison
│   ├── Bar
│   ├── Column
│   └── Dual Axis
├── Trend
│   ├── Line
│   ├── Area
│   └── Combo
├── Composition
│   ├── Pie
│   ├── Donut
│   └── Treemap
├── Distribution
│   ├── Histogram
│   ├── Box Plot
│   └── Scatter
└── Advanced
    ├── Pivot Table
    ├── Pivot Chart
    └── Sankey
```

设计细节：

- 每个图表类型用小预览图；
- 推荐图表放在最上方；
- 不适合当前字段的图表置灰；
- 置灰时说明原因，例如 “需要至少 1 个 measure”；
- 切换图表后，不要静默删除字段；
- 不匹配字段放进 unmatched fields 区域。

### 模块 B：Theme Studio / 主题模块

DataWind 风格特别适合强调主题系统。

```text
Theme Studio
├── Mode
│   ├── Light
│   ├── Dark
│   ├── Dashboard Dark
│   └── Professional Blue
├── Palette
│   ├── Default
│   ├── Soft
│   ├── Business
│   └── High Contrast
├── Typography
│   ├── Title
│   ├── Axis
│   └── Label
└── Chart Elements
    ├── Legend
    ├── Tooltip
    ├── Gridline
    └── Data Label
```

注意：主题模块不要第一屏全部展开。应该分层：

```text
minimalist：Light / Dark
standard：Light / Dark + Palette
professional：完整 Theme Studio
```

### 模块 C：Interaction Layer / 图表交互层

产品级 VBI 可以增加交互配置：

```text
Interactions
├── Tooltip
│   ├── Basic tooltip
│   ├── Rich tooltip
│   └── Tooltip fields
├── Selection
│   ├── Click select
│   ├── Brush select
│   └── Select to filter
├── Drill
│   ├── Enable drill down
│   └── Drill path
├── Zoom / Pan
│   ├── Enable zoom
│   └── Enable pan
└── Annotation
    ├── Reference line
    ├── Marker
    └── Note
```

这个模块会让页面更像专业 BI，而不是普通图表 demo。

## 5.3 DataWind 风格页面结构示意

```text
┌──────────────────────────────────────────────────────────────┐
│ Toolbar: Chart Gallery | Theme | Interaction | Fullscreen    │
├───────────────┬──────────────────────────────┬───────────────┤
│ Fields        │ Chart Workspace              │ Visual Config │
│               │ ┌──────────────────────────┐ │ Chart Type    │
│ Dimensions    │ │                          │ │ Theme Studio  │
│ Measures      │ │      Rich Chart Area      │ │ Interaction   │
│               │ │                          │ │ Annotation    │
│               │ └──────────────────────────┘ │ Tooltip       │
│               │ Insight / Annotation Layer   │               │
└───────────────┴──────────────────────────────┴───────────────┘
```

## 5.4 什么时候采用 DataWind 风格

适合：

- 图表类型多；
- 需要主题、视觉配置、交互配置；
- 希望页面看起来更现代、更产品化；
- 图表不仅用于调试，还要用于展示和分析。

不适合：

- MVP 起步阶段；
- 功能还没跑通时；
- 字段映射和查询逻辑还不稳定时。

# 6. Superset 风格模块：标准 no-code 配置面板

## 6.1 Superset 的核心气质

Superset 的 Explore 类页面更偏 “配置面板 + 图表预览”。它适合把复杂图表配置拆成清楚的表单区。

最值得学习的是：

```text
Query 和 Customize 分开。
```

也就是：

```text
Query：数据怎么来，怎么聚合，怎么过滤
Customize：图表怎么显示，颜色、坐标轴、tooltip、label 怎么配置
```

这个思想非常适合 VBI 的 standard 页面。

## 6.2 Superset 风格适合 VBI 的模块

### 模块 A：Query Panel

```text
Query
├── Dataset / Data Source
├── Chart Type
├── Dimensions / Group by
├── Measures / Metrics
├── WHERE Filters
├── HAVING Filters
├── Sort
└── Limit
```

这个区域回答的是：

```text
图表的数据结果是怎么算出来的？
```

### 模块 B：Customize Panel

```text
Customize
├── Theme
├── Color Scheme
├── Legend
├── Axis
│   ├── X Axis Label
│   ├── Y Axis Label
│   └── Axis Format
├── Tooltip
├── Data Label
└── Layout
```

这个区域回答的是：

```text
这个图表应该怎么展示？
```

### 模块 C：Run Query / Auto Update

对于小数据，用户可以实时更新。对于大数据，最好支持手动运行。

```text
Update Mode
├── Auto update: 每次配置变动自动刷新
└── Manual run: 修改后点击 Run Query
```

UI 上可以这样表现：

```text
[ Auto update: ON ▼ ]    [ Run Query ]
```

或者：

```text
You have unapplied changes.  [Apply Changes]
```

## 6.3 Superset 风格页面结构示意

```text
┌──────────────────────────────────────────────────────────────┐
│ Header: Dataset | Chart Type | Save | Run Query              │
├──────────────────────┬───────────────────────────────────────┤
│ Query / Customize    │ Chart Preview                         │
│                      │                                       │
│ [Query] [Customize]  │ ┌───────────────────────────────────┐ │
│                      │ │                                   │ │
│ Dataset              │ │              Chart                │ │
│ Dimensions           │ │                                   │ │
│ Measures             │ └───────────────────────────────────┘ │
│ WHERE                │                                       │
│ HAVING               │ Result: 500 rows · updated 2s ago     │
│ Limit                │                                       │
│                      │                                       │
│ [Run Query]          │                                       │
└──────────────────────┴───────────────────────────────────────┘
```

## 6.4 什么时候采用 Superset 风格

适合：

- standard 完整 UI；
- 配置项较多；
- 用户不一定熟悉拖拽；
- 需要清晰区分数据计算和视觉展示；
- 数据查询可能较慢，需要 Run Query。

不适合：

- 追求强交互拖拽体验的 professional 编辑器；
- 极简 view-only 页面。

# 7. Hex 风格模块：Notebook-to-App / 编辑态与展示态分离

## 7.1 Hex 的核心气质

Hex 类产品的核心是：

```text
分析过程可以复杂，但发布给用户看的结果必须干净。
```

它强调从分析、图表、交互控件到数据应用的转换。对 VBI 来说，最值得学习的是：

```text
Edit Mode 和 View Mode 分离。
```

## 7.2 Hex 风格适合 VBI 的模块

### 模块 A：Edit Mode

编辑态可以显示完整工具：

```text
Edit Mode
├── Field Panel
├── Config Panel
├── Filter Editor
├── Chart Workspace
├── Debug Panel 可选
└── Status Footer
```

编辑态服务于“构建图表”。

### 模块 B：View Mode

展示态应该隐藏复杂配置，只保留用户真正需要操作的控件：

```text
View Mode
├── Title / Description
├── Control Strip
│   ├── Date Range
│   ├── Region
│   ├── Category
│   └── Metric
├── Main Chart
├── Insight Cards
└── Data Preview / Detail Table
```

展示态服务于“阅读和使用图表”。

### 模块 C：Control Strip

View Mode 不应该显示 WHERE / HAVING 的底层结构，而应该显示业务化控件：

```text
Controls
├── Date Range: Last 90 days
├── Region: All
├── Category: Furniture
├── Metric: Sales
└── Refresh
```

底层可以映射到 filter 或 metric，但 UI 语言应该更像业务应用。

### 模块 D：Insight Cards

Hex 风格页面经常有总结型模块。VBI 可以加：

```text
Insight Cards
├── Total Sales
├── Top Category
├── Growth Rate
└── Data Rows
```

这个模块让页面从“工具”变成“分析应用”。

## 7.3 Hex 风格页面结构示意

```text
View Mode / App Mode

┌──────────────────────────────────────────────────────────────┐
│ Sales Explorer                                  Export Share │
│ Analyze sales by category, region, and time.                 │
├──────────────────────────────────────────────────────────────┤
│ Date Range [Last 90 days ▼] Region [All ▼] Metric [Sales ▼]  │
├──────────────────────────────────────────────────────────────┤
│ ┌──────────────────────────────────────────────────────────┐ │
│ │                     Main Chart                           │ │
│ └──────────────────────────────────────────────────────────┘ │
│                                                              │
│ Insight Cards                                                │
│ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐           │
│ │ Total Sales  │ │ Top Region   │ │ Growth       │           │
│ └──────────────┘ └──────────────┘ └──────────────┘           │
│                                                              │
│ Detail Table Preview                                         │
└──────────────────────────────────────────────────────────────┘
```

## 7.4 什么时候采用 Hex 风格

适合：

- 需要 view/edit 模式；
- 图表构建完成后要展示给别人看；
- 页面需要更像数据应用，而不只是编辑器；
- 想让非技术用户也能使用。

不适合：

- 只做内部调试器；
- 所有操作都必须实时暴露给开发者的页面。

# 8. 三种页面风格：Minimalist / Standard / Professional

## 8.1 Minimalist：最小可用 UI

目标：功能能跑通，页面干净，不复杂。

适合功能清单：

- 加载 demo 数据；
- 展示字段；
- 添加 dimension；
- 添加 measure；
- 切换 chart type；
- 修改 limit；
- 添加 WHERE filter；
- 渲染图表；

# 10. 八大核心模块怎么设计

## 10.1 App Shell / 页面框架

App Shell 是页面外壳。它负责整体布局，不负责具体配置。

应该包含：

- 明暗主题；
- 语言；
- 全屏状态；
- 编辑 / 查看模式；
- 全局 loading / error；
- 页面宽高稳定性。

设计建议：

```text
App Shell
├── 顶部固定 Toolbar
├── 中间三栏 Workbench
└── 可选底部状态栏
```

App Shell 的美观重点：

- 背景不要纯白，使用浅灰蓝；
- 面板之间有 12–16px 间距；
- 面板用统一圆角；
- 不要让任何状态导致页面高度突然塌陷；
- 全屏时隐藏外部导航，但保留图表操作。

## 10.2 Top Toolbar / Chart Toolbar

Toolbar 应该放高频、全局、不会让用户困惑的操作。

推荐顺序：

```text
左侧：数据源状态 / 页面标题
中间：Chart Type / Theme / Locale / Limit
右侧：Undo / Redo / Refresh / Fullscreen / View-Edit / Save
```

不要把所有设置都塞进 Toolbar。低频设置放 Config Panel。

错误示例：

```text
Toolbar 里塞满 20 个按钮，每个按钮都是 icon，没有文字解释。
```

正确示例：

```text
Toolbar 只放 6–9 个最重要操作，其余放 More 或 Settings。
```

## 10.3 Field Panel / 数据字段面板

字段面板的目标是让用户快速理解数据结构。

推荐结构：

```text
Fields
[ Search fields... ]

Dimensions
  Category        string
  Region          string
  Order Date      date

Measures
  Sales           number
  Profit          number
  Quantity        number
```

视觉细节：

- 字段卡片高度 32–36px；
- icon + 字段名 + 类型 badge；
- 长字段名省略；
- hover 后出现 Add / Drag handle；
- 搜索无结果时显示 empty state；
- 维度和指标用颜色区分，但颜色要轻。

## 10.4 Config Panel / Shelves / Slot Overlay

这是用户配置图表的核心区域。

两种主流形态：

### 形态 A：Standard Shelves

```text
Dimensions  [Category ×] [Region ×]
Measures    [SUM(Sales) ▼ ×]
WHERE       [Region = East ×]
HAVING      [SUM(Sales) > 1000 ×]
```

适合 standard 页面。

### 形态 B：Professional Slot Overlay

```text
Visual Mapping
├── X Axis       Drop dimension here
├── Y Axis       Drop measure here
├── Color        Drop field here
├── Size         Drop measure here
├── Label        Drop field here
└── Tooltip      Drop fields here
```

适合 professional 页面。

Slot 状态：

| 状态     | UI 表现                            |
| -------- | ---------------------------------- |
| 空       | 虚线边框 + placeholder             |
| 可拖入   | 蓝色边框 + 浅蓝背景                |
| 不可拖入 | 红色边框 + 原因提示                |
| 已填     | token + 删除按钮 + 菜单            |
| 聚合字段 | 显示 `SUM(Sales)` 这种业务可读文案 |

## 10.5 Filter UI

WHERE 和 HAVING 必须分开。

```text
Filters

Raw Rows: WHERE
  Region = East / West
  Order Date between 2024-01-01 and 2024-12-31

Aggregated Results: HAVING
  SUM(Sales) > 1000
```

设计原因：

- WHERE 过滤原始行；
- HAVING 过滤聚合结果；
- 两者混在一起会让用户误解计算逻辑。

Filter row 结构：

```text
[field] [operator] [value] [delete]
```

Root operator：

```text
Match [ALL / ANY] of the following filters
```

等价于：

```text
AND / OR
```

但对普通用户更友好。

## 10.6 Chart Workspace

Chart Workspace 是主视觉区域。它必须稳定、清楚、不会跳。

推荐结构：

```text
Chart Workspace
├── Header
│   ├── Chart Title
│   ├── Updated time
│   └── More actions
├── Filter Chips
├── Body
│   ├── Empty
│   ├── Loading
│   ├── Error
│   └── Rendered Chart
└── Status Bar
    ├── rows
    ├── marks
    └── warning
```

关键要求：

- loading 不要让图表区域高度塌陷；
- empty 要告诉用户下一步做什么；
- error 要提供 Retry 和 View detail；
- rendered 状态要显示 rows / marks / updated time；
- 图表容器应该有 min-height。

## 10.7 VSeedRender / 渲染边界

这里不讨论代码，只讨论 UI 责任边界。

VSeedRender 在页面设计上应该被理解为：

```text
一个稳定的图表渲染边界。
```

它不应该承担：

- 字段选择；
- 筛选器编辑；
- 数据源上传；
- 页面布局；
- 配置面板。

它应该承担：

- 渲染图表；
- 显示渲染错误；
- 保证容器尺寸稳定；
- 切换图表时平滑过渡；
- 渲染失败时不拖垮整个页面。

## 10.8 Data Source / 数据加载

数据入口应该像产品功能，而不是临时按钮。

推荐结构：

```text
Data Source
├── Load Demo
├── Upload CSV
├── Switch Source
├── Refresh Schema
└── Source Status
```

状态卡片：

```text
● Connected  Sample Orders
12 fields · 10,000 rows
[Refresh schema] [Change]
```

CSV 上传状态：

| 状态   | UI                  |
| ------ | ------------------- |
| 未上传 | Drop CSV here       |
| 上传中 | Uploading...        |
| 解析中 | Inferring schema... |
| 成功   | 12 fields detected  |
| 失败   | Parse error + retry |

# 11. 状态设计：让页面稳定、不跳、不迷惑

## 11.1 页面状态机

```text
Uninitialized
  ↓
Initializing
  ↓
No Data Source
  ↓
Schema Loaded
  ↓
Empty Chart Config
  ↓
Building Query
  ↓
Loading Result
  ↓
Rendering Chart
  ↓
Rendered
```

错误分支：

```text
Data Error
Query Error
Render Error
Empty Result
```

## 11.2 每种状态怎么画

| 状态         | UI 设计                                 |
| ------------ | --------------------------------------- |
| initializing | 页面骨架屏，不要白屏                    |
| no data      | 中间空状态，引导 Load Demo / Upload CSV |
| empty chart  | 提示添加 dimension 和 measure           |
| loading      | 图表 skeleton + 顶部细进度条            |
| empty result | 显示 “No rows match current filters”    |
| query error  | Error card + Retry + View query detail  |
| render error | Chart 区域 fallback，不影响整个页面     |
| rendered     | 图表 + metadata + filter chips          |

## 11.3 错误状态的好坏对比

差：

```text
Error
```

好：

```text
Could not render this chart
The current chart type requires at least one measure.

[Add measure] [Change chart type] [View detail]
```

# 12. 交互与性能：顺滑比炫酷更重要

## 12.1 拖拽交互

拖拽时要有明确反馈：

```text
字段被拖起 → 可放置 slot 高亮 → 放入后 token 出现 → 图表更新
```

不要出现：

```text
拖了字段，但不知道哪里能放。
```

## 12.2 配置更新

小数据：

```text
改配置 → 自动更新图表
```

大数据：

```text
改配置 → 显示 unapplied changes → 点击 Run Query
```

## 12.3 动效原则

动效只用于解释状态变化，不用于装饰。

推荐：

- hover：120–150ms；
- panel 展开：180–220ms；
- chart fade in：150–200ms；
- drag shadow：轻微；
- skeleton loading：稳定不闪烁。

避免：

- 大面积弹跳；
- 图表区域频繁重排；
- loading 时整页闪烁；
- 每次输入都全页面刷新。

# 13. 常见失败设计案例

## 13.1 把所有配置堆成一个大表单

失败原因：

```text
用户不知道哪些是数据计算，哪些是视觉样式。
```

改法：

```text
Query / Customize 分离。
```

## 13.2 字段面板没有 dimension / measure 分组

失败原因：

```text
用户不知道哪些字段可以做 X，哪些字段适合做 Y。
```

改法：

```text
字段按角色分组，并用 icon / color / badge 区分。
```

## 13.3 WHERE 和 HAVING 混在一起

失败原因：

```text
过滤原始行和过滤聚合结果的语义不同，混在一起会造成错误理解。
```

改法：

```text
Raw Rows: WHERE
Aggregated Results: HAVING
```

## 13.4 loading 导致布局跳动

失败原因：

```text
图表区域高度忽高忽低，用户感觉页面不稳定。
```

改法：

```text
Chart Workspace 设置稳定 min-height，loading 用 skeleton。
```

## 13.5 工具栏按钮太多

失败原因：

```text
用户不知道哪个操作最重要。
```

改法：

```text
Toolbar 只保留高频操作，低频放进 More / Settings。
```

## 13.6 页面太像“调试工具”

失败原因：

```text
普通用户看到 builder、dsl、vseed、schema 这些词会困惑。
```

改法：

```text
普通模式隐藏技术词，Debug 信息放进 Developer 抽屉。
```

# 最后一条原则

不要让 agent 只记住一个模板。  
要让 agent 记住这句话：

> 图表构建 UI 的本质，是把“数据字段”清楚地映射到“视觉表达”，并在每一步给用户稳定、可理解、可恢复的反馈。
